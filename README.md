# Business Oriented Bug Optimization & Diagnostic Deployment sYstem

Next.js app for ingesting, searching, and analyzing a large ticket backlog.

## Stack

- Node.js + Next.js App Router
- Zod contracts for API validation
- Drizzle ORM + PostgreSQL
- Material UI components
- Chart.js dashboards
- Vitest + Testcontainers for endpoint tests

## Getting Started

1. Copy env file:

```bash
cp .env.example .env
```

2. Start PostgreSQL with Docker Compose:

```bash
docker compose up -d db
```

This project uses `localhost:6429` for local Postgres by default.

3. Push schema:

```bash
pnpm run db:push
```

4. Run app:

```bash
pnpm run dev
```

## Scripts

- `pnpm run dev` - starts local Postgres (`docker compose up -d db`) and runs Next.js
- `pnpm run build` - production build
- `pnpm run start` - start built app
- `pnpm run lint` - lint project
- `pnpm run db:push` - apply Drizzle schema to DB
- `pnpm run db:generate` - generate SQL migrations
- `pnpm run test` - run integration tests

## AI Agent Runtime

The worker remains the orchestrator for queued step executions. By default, AI-backed
steps launch GitHub Copilot coding agent runs. To switch those steps to a sandbox
runtime instead, set:

```bash
AI_AGENT_PROVIDER=sandbox
SANDBOX_AGENT_BASE_URL=http://localhost:4000
SANDBOX_AGENT_REPOSITORY=owner/repo
APP_BASE_URL=http://localhost:3000
BOBODDY_API_KEY=key_for_api_authentication
```

In sandbox mode, the app POSTs an agent run request to
`$SANDBOX_AGENT_BASE_URL/agent-runs` including:

- repo/ref/instructions for the run
- step and ticket identifiers
- a callback target pointing at the existing webhook routes under `/api/webhooks/*`
- the API key the sandbox should use when it submits the completion payload

The sandbox should execute the repo work and then `PUT` its result JSON back to the
provided callback URL.

When you want the sandbox API to run as part of the existing worker service, enable:

```bash
SANDBOX_SERVICE_ENABLED=true
SANDBOX_AGENT_PORT=4000
SANDBOX_RUNNER_MODE=docker_compose
SANDBOX_DOCKER_COMPOSE_FILE=./docker-compose.sandbox.yml
SANDBOX_DOCKER_COMPOSE_SERVICE=agent-runner
SANDBOX_SOURCE_REPO_PATH=/absolute/path/to/this/repo
```

That starts a small HTTP service inside the worker process with:

- `POST /agent-runs` to create a run
- `GET /agent-runs/:runId` to inspect runner status

The worker remains the queued-step orchestrator. The sandbox service is a sibling
module that owns sandbox execution concerns.

The compose-backed runner uses [docker-compose.sandbox.yml](/Users/connor/Fiddle/voc-viewer/docker-compose.sandbox.yml) and
[Dockerfile.sandbox-runner](/Users/connor/Fiddle/voc-viewer/Dockerfile.sandbox-runner). The container entrypoint is
[run-sandbox-agent-task.ts](/Users/connor/Fiddle/voc-viewer/src/modules/ai/infra/run-sandbox-agent-task.ts).

The sandbox runner now uses the Vercel AI SDK directly for structured generation. It uses
the exact `customInstructions` generated by the existing step use cases as the model prompt,
then adds compact repository context from the checked-out workspace before generating a
structured JSON payload for the step.

Relevant env:

```bash
SANDBOX_AGENT_MODEL_PROVIDER=github-models
GITHUB_MODELS_API_KEY=your_github_models_token
SANDBOX_AGENT_MODEL=openai/gpt-4.1-mini
```

To use a local Codex CLI instance instead of GitHub Models:

```bash
SANDBOX_AGENT_MODEL_PROVIDER=codex-cli
SANDBOX_AGENT_MODEL=gpt-5.3-codex
OPENAI_API_KEY=
```

The Codex path uses the `ai-sdk-provider-codex-cli` package and spawns `codex exec`
inside the checked-out sandbox workspace. Authentication comes from `codex login`
(`~/.codex/auth.json`) or `OPENAI_API_KEY`. The compose-backed runner now mounts
`${HOME}/.codex` into the container and forwards `OPENAI_API_KEY` so local Codex auth
works in either mode.

If generation fails, the runner sends a schema-compatible `agent_error` payload instead.
